# 📱 전화번호 SMS 인증 & 소셜 계정 통합 — 인수인계 문서

> 작성일: 2026-07-06
> 목적: 카카오·구글 로그인 사용자를 **전화번호를 키로 동일인으로 묶기** 위해 SMS 인증을 도입.
> 이 문서만 읽으면 무슨 기능인지 / 어디를 고치면 되는지 알 수 있도록 정리했습니다.

---

## 1. 한 줄 요약

카카오·구글 둘 다 로그인 수단으로 쓰되, **최초 1회 전화번호 SMS 인증**을 받아서
같은 전화번호면 같은 사람으로 계정을 통합합니다. (예: 카카오로 가입했던 사람이 나중에 구글로 로그인해도, 전화번호가 같으면 기존 계정으로 연결)

문자 발송은 **Solapi(솔라피)** 라는 문자 API 서비스를 사용합니다.

---

## 2. 사용자 입장에서의 흐름

```
카카오/구글 로그인
   ↓
약관 동의 (기존 화면)
   ↓
📱 전화번호 입력  →  "인증번호 받기" (문자 발송)
   ↓
📱 숫자 키패드로 6자리 인증번호 입력  →  자동 확인
   ↓
   ├─ 이 번호로 가입한 기존 계정이 있으면  → 기존 계정으로 통합(로그인)
   └─ 없으면                              → 이 번호를 현재 계정에 연결
   ↓
홈 화면 진입 (이후로는 인증 화면 안 뜸)
```

- 한 번 인증하면 `users` 문서에 `phoneVerified: true`가 저장되어 **다시는 안 뜹니다.**
- 관리자(어드민) 계정은 이 인증을 면제합니다.

## 3. 파일 구조 (어디에 무엇이 있는지)

### 서버 (Cloud Functions) — 문자 발송/검증
| 파일 | 역할 |
|------|------|
| `functions/otp/phoneOtp.js` | **핵심.** 인증번호 발송(`requestPhoneOtp`), 검증(`verifyPhoneOtp`) 두 개의 HTTP 함수. Solapi로 문자 전송. |
| `functions/index.js` | 위 두 함수를 export (파일 하단 "전화번호 SMS 인증" 주석 부분) |
| `functions/package.json` | `solapi` 패키지 의존성 추가됨 (버전 `^5.5.4` — ⚠️ 6.x는 API가 달라서 안 됨) |

### 프론트 (React)
| 파일 | 역할 |
|------|------|
| `src/pages/auth/PhoneVerifyPage.jsx` | **인증 화면.** 전화번호 입력 + 숫자 키패드로 6자리 코드 입력 UI + 3분 타이머/재전송 |
| `src/services/phoneOtpService.js` | 위 서버 함수를 호출하는 함수(`requestPhoneOtp`, `verifyPhoneOtp`) + 전화번호를 `+82` 형식으로 바꾸는 `toE164Kr` |
| `src/routes/AppRoutes.jsx` | `RequirePhone` 게이트 — `phoneVerified`가 아니면 위 인증 화면을 강제로 띄움 |
| `src/context/AuthContext.jsx` | 로그인 사용자 정보 로드. `getUserProfileByUid`로 통합된 계정까지 찾아냄 |

### 이미 있던(재활용한) 파일 — 계정 통합 로직
| 파일 | 역할 |
|------|------|
| `src/services/phoneService.js` | `linkPhoneToUid`(번호↔계정 연결), `getPrimaryUidByPhone`(번호로 기존계정 찾기) |
| `src/services/userService.js` | `linkSocialToExistingUser`(소셜계정을 기존계정에 통합) |

API Key/Secret은 보안상 코드에 넣지 않고 Firebase 시크릿으로 관리합니다.
**다른 PC에서 배포할 때는 시크릿이 이미 서버에 등록돼 있으므로 재등록 불필요**하지만,
혹시 다시 등록해야 하면:
```bash
firebase functions:secrets:set SOLAPI_API_KEY --project halle-bf789
# 물어보면 값 입력: NCS4CF8Z3BJ7IBWD

firebase functions:secrets:set SOLAPI_API_SECRET --project halle-bf789
# 물어보면 값 입력: (솔라피 콘솔의 API Secret)
```

## 6. 데이터 구조 (Firestore)

- **`phone_verifications`** (신규 컬렉션): 인증번호 발송 기록.
  `{ phone, code, purpose, expiresAt, verified, attempts, isTest, createdAt }`
  - 인증번호 유효 3분, 오입력 5회 제한, 같은 번호 1시간 5회 발송 제한.
- **`users`**: 기존 컬렉션에 아래 필드 사용(이미 있던 필드)
  - `phoneE164`: 전화번호(`+821012345678` 형식)
  - `phoneVerified`: 인증 완료 여부(`true`면 인증 화면 안 뜸)
  - `linkedSocialUid`: 통합된 소셜 계정 UID
- **`phones/{전화번호}`**: 전화번호 ↔ 계정 매핑 (`primaryUid`, `linkedUids[]`)

## 7. 이어서 개발할 때 알아둘 점 / 주의사항

1. **기존 사용자도 다음 로그인 때 인증 화면이 한 번 뜹니다.**
   (기존 계정엔 `phoneVerified`가 없기 때문. 전화번호를 새로 수집·통합하려는 의도)
   - 만약 기존 사용자는 건너뛰게 하고 싶으면 `AppRoutes.jsx`의 `RequirePhone` 함수에서
     조건을 조정하면 됩니다. (예: 특정 가입일 이전 사용자 면제)

2. **테스트 번호** (앱스토어 심사용): `01099991000` ~ `01099991005` 는
   실제 문자를 안 보내고 화면에 인증번호가 그대로 표시됩니다. (`phoneOtp.js`의 `TEST_PHONE_RANGE`)

3. **인증 화면 UI**는 `PhoneVerifyPage.jsx`에 styled-components로 작성.
   숫자 키패드는 plhouse 프로젝트의 코드입력 UI를 참고했습니다. 라이트/다크 모드 모두 대응.

4. **문자 문구 변경**: `functions/otp/phoneOtp.js`의 `smsText` 부분
   (`[할래말래] 인증번호는 ... 입니다.`) 수정 후 함수 재배포.

5. **보안 참고**: 현재 Firestore 보안 규칙이 느슨한 상태라, 클라이언트가 `phoneVerified`를
   직접 쓸 수 있는 구조입니다. 실서비스 강화 시 인증 검증을 서버(verifyPhoneOtp)에서
   토큰 발급 → 서버에서 통합 처리하는 방식으로 조이는 것을 권장합니다.
   (참고 구현: ieum 프로젝트의 `linkSocialToPhone` 서버 함수 + verifiedToken 방식)
